Document 01 — PRD
Product Requirements Document
Full specification of what Shadow Chats is, why it exists, who uses it, and what it must do.

🎯 Problem Statement

People often don't know what to reply — especially under social pressure, with a crush, in a heated argument, or when they want to sound perfect. Typing and retyping takes time. Ghosting happens because replies feel too hard. Shadow Chats solves this by generating context-aware, tone-matched reply suggestions instantly.

👤 Target Users

Primary: 18–28 year olds in India using WhatsApp and Instagram daily. Secondary: professionals who need help with formal/work chats. Tertiary: anyone who struggles with social anxiety or language barriers while chatting.

Document 03 — HLD
High-Level Design
Service-level design, API contracts, communication patterns, and infrastructure decisions.

Service Breakdown

🔐 Auth Service

Responsibilities: User registration, login, JWT issue/refresh, Google OAuth, rate limit enforcement per user tier.

Endpoints: POST /auth/register, POST /auth/login, POST /auth/refresh, GET /auth/me

📷 OCR Service

Responsibilities: Accept screenshot upload, extract text using Google Vision (primary) or Tesseract (fallback), return structured conversation array with speaker labels.

Endpoints: POST /ocr/extract — returns {messages: [{speaker, text, timestamp}]}

🤖 Reply Service

Responsibilities: Accept conversation context + tone preference, call AI engine, return reply suggestions + emotion score. Core of the product.

Endpoints: POST /reply/generate, POST /reply/regenerate, GET /reply/history

🎭 Emotion Service

Responsibilities: Run DistilRoBERTa on input text, return emotion label (anger/joy/sadness/surprise/fear/neutral) + intensity score 0–100 for the Reply Meter.

Endpoints: POST /emotion/analyze — returns {label, score, breakdown}

👤 User Service

Responsibilities: Profile management, persona settings, tone preferences, subscription status, daily quota tracking.

Endpoints: GET/PUT /user/profile, GET /user/quota, PUT /user/persona

⚙️ Config Service

Responsibilities: Store and serve user's AI engine preference (Groq / Gemini / Self-hosted). Validate API keys. Handle fallback logic when quota exhausted.

Endpoints: GET/PUT /config/ai-engine, POST /config/test-key

AI Engine Abstraction Layer

🔄 Engine Router Logic

The backend uses an AIEngineRouter class that wraps all AI providers. User preference is stored per-account. The router automatically falls back if quota is hit:

class AIEngineRouter:
  engines = ["groq", "gemini", "custom"]
  
  async def generate(context, tone, engine_pref):
    engine = engine_pref or user.preferred_engine
    
    try:
      return await self._call(engine, context, tone)
    except QuotaExceeded:
      fallback = self._next_engine(engine)
      return await self._call(fallback, context, tone)
  
  def _build_prompt(context, tone):
    system = f"""You are a chat reply assistant.
    Generate {count} replies in {tone} tone.
    Context language: {language}.
    Reply ONLY with JSON array of reply strings.
    Max 2 sentences per reply. Sound natural."""
    return system, format_context(context)

This means a user can pick "Groq" in settings but if their free quota runs out mid-session, the app silently switches to Gemini without breaking the experience.

# ===== CORE MODELS =====
class ReplyRequest(BaseModel):
    context: List[Message]          # conversation messages
    tone: ToneEnum                  # flirty|funny|formal|savage|caring|chill
    count: int = 3                  # how many replies
    language: str = "en"            # en|hi|bn|hinglish
    engine: Optional[EngineEnum]    # override user's default

class Message(BaseModel):
    speaker: Literal["me", "them"]
    text: str
    timestamp: Optional[str]

class ReplyResponse(BaseModel):
    replies: List[ReplyOption]
    emotion: EmotionResult
    engine_used: str
    tokens_used: int
    session_id: str

class ReplyOption(BaseModel):
    text: str
    score: int          # confidence/quality 0-100

class EmotionResult(BaseModel):
    label: str          # anger|joy|sadness|fear|surprise|neutral
    score: int          # intensity 0-100
    breakdown: dict     # {"anger": 12, "joy": 65, ...}

# ===== AI ENGINE ROUTER =====
class AIEngineRouter:
    def __init__(self):
        self.engines = {
            "groq": GroqEngine(),
            "gemini": GeminiEngine(),
            "custom": CustomModelEngine()
        }
        self.fallback_order = ["groq", "gemini", "custom"]

    async def generate(self, req: ReplyRequest, user: User) -> ReplyResponse:
        engine_name = req.engine or user.preferred_engine
        for attempt_engine in [engine_name] + self.fallback_order:
            try:
                engine = self.engines[attempt_engine]
                if await engine.is_available():
                    result = await engine.generate(req)
                    result.engine_used = attempt_engine
                    return result
            except (QuotaExceededException, RateLimitException):
                continue
        raise AllEnginesExhaustedException()

Hybrid App Architecture (React Native + Expo)

♻️ Why React Native Web (Hybrid Approach)

Instead of building Web in React, then Android in Kotlin, then iOS in Swift — one single React Native + Expo codebase compiles to all three. Write once:

  • expo build:web → deploys to any hosting (Vercel/Netlify/Railway)
  • eas build --platform android → builds APK / AAB for Play Store
  • eas build --platform ios → builds IPA for App Store

Platform-specific code is isolated in /src/platform/ — e.g. camera access, clipboard, share sheet.

Emotion Detection Module (Own Model Approach)

🎭 Building Your Own Emotion Classifier

This is the "I built something myself" part of the portfolio story. Use HuggingFace's pre-trained j-hartmann/emotion-english-distilroberta-base model exposed via a Flask API:

# emotion_service.py — Your own Python microservice
from transformers import pipeline
from fastapi import FastAPI

app = FastAPI()

# Load once at startup
emotion_classifier = pipeline(
    "text-classification",
    model="j-hartmann/emotion-english-distilroberta-base",
    top_k=None  # get all emotion scores
)

@app.post("/analyze")
async def analyze_emotion(text: str):
    results = emotion_classifier(text)[0]
    # Returns: [{"label":"joy","score":0.87}, {"label":"anger","score":0.05}, ...]
    
    top_emotion = max(results, key=lambda x: x['score'])
    intensity = int(top_emotion['score'] * 100)
    
    return {
        "label": top_emotion['label'],
        "score": intensity,
        "breakdown": {r['label']: round(r['score']*100) for r in results}
    }

This Flask/FastAPI service runs on the same server as your backend (or on HuggingFace Spaces for free). The Reply Generator then uses the emotion score to calibrate reply tone.

Document 08 — Browser Extension
WhatsApp & Instagram Extension
Chrome Extension (Manifest V3) that injects a Shadow Chats button directly into WhatsApp Web and Instagram DMs.

🔧 Extension Architecture

manifest.json: Declares permissions for whatsapp.com and instagram.com. Uses content_scripts and background service_worker.

content_script.js: Injected on page load. Watches DOM for chat input area. Injects the Shadow Chats button.

popup.html: The extension popup with settings and quick generate.

background.js: Service worker. Handles API calls to Shadow Chats backend. Manages auth token.

📌 DOM Injection Strategy

WhatsApp Web: Find the footer div with [data-tab="10"]. Inject a button div after the emoji button.

Instagram DMs: Find the input area [placeholder="Message..."]. Insert button in the parent row.

Context Reading: Read all .message-in and .message-out elements (last 15 messages) from DOM to build context.

engine_status:{name} → up/down flag (TTL: 5min)
